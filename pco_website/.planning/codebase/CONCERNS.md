# Codebase Concerns

**Analysis Date:** 2026-03-05

## Tech Debt

**Frontend is a scaffolded placeholder:**
- Issue: The entire Next.js frontend is still the default `create-next-app` template. It contains no PCO-specific UI, uses the Next.js logo, and links to Vercel/Next.js marketing pages.
- Files: `pco_website/app/page.tsx`, `pco_website/app/layout.tsx`
- Impact: There is no working frontend; all backend API endpoints exist with no consumer. Page metadata reads "Create Next App" / "Generated by create next app".
- Fix approach: Replace `page.tsx` with actual PCO pages. Update `layout.tsx` metadata (title, description). Build authentication flow, member dashboard, public pages.

**`/v1/public/info` endpoint returns hardcoded TODO strings:**
- Issue: The public info endpoint returns `"TODO"` string literals for `history`, `philanthropy`, `join_now_url`, `rush_week.dates`, `rush_week.details`, `contact.rush_chairs`, and `contact.t6`.
- Files: `pco_backend/app/api/v1/public.py`
- Impact: Any frontend or API consumer calling `/v1/public/info` receives placeholder data. Unblocking requires either wiring it to the DB content system or populating it.
- Fix approach: Route these fields to the existing `OrgContent` DB table (sections `history`, `philanthropy`, `contacts`) and the `RushInfo` table, which already store this data.

**`admin/settings.py` router is an empty placeholder:**
- Issue: `app/api/v1/admin/settings.py` is registered in the main router at `/v1/admin/settings` but contains zero endpoints — only a comment noting intent was migrated elsewhere.
- Files: `pco_backend/app/api/v1/admin/settings.py`, `pco_backend/app/api/router.py`
- Impact: Registered but dead route prefix. Creates confusion in API docs (Swagger shows an empty `admin-settings` tag). Unnecessary include in router.
- Fix approach: Either add settings endpoints or remove the router include from `app/api/router.py`.

**`models/role.py` is an empty file:**
- Issue: `app/models/role.py` is a completely empty file (0 bytes). It is imported nowhere but exists in the models directory.
- Files: `pco_backend/app/models/role.py`
- Impact: Dead file adds confusion — suggests a `Role` model was planned but never built. Role is currently stored as a plain string column on `User`.
- Fix approach: Delete the file, or implement a proper `Role` enum/model if role constraint enforcement is needed at the DB layer.

**Role validation is split across Pydantic and runtime code:**
- Issue: Valid roles are defined as `ALL_ROLES = ["member", "admin"] + OFFICER_ROLES` in `models/user.py` and duplicated as a `Literal` type in `schemas/user.py`. Any new role addition requires updating both places.
- Files: `pco_backend/app/models/user.py`, `pco_backend/app/schemas/user.py`
- Impact: The `update_member_role` service also performs a runtime check against `ALL_ROLES`, creating a third place to update. Pydantic validation enforces the Literal but the service check is redundant.
- Fix approach: Generate `_AllRolesLiteral` dynamically from `ALL_ROLES` using `Literal` from `typing` with unpacking, removing the manual duplication. Remove redundant runtime check in `user_service.py`.

**Temp password delivery has no fallback tracking:**
- Issue: When a member is created via `POST /v1/admin/users/`, a temporary password is generated and sent via background email task. If the email send fails (swallowed silently), the admin has no way to retrieve the temp password — it is not stored anywhere.
- Files: `pco_backend/app/api/v1/admin/users.py`, `pco_backend/app/services/email_service.py`, `pco_backend/app/services/user_service.py`
- Impact: If SMTP is misconfigured or delivery fails, new member accounts become inaccessible with no recovery path other than admin manually resetting the password — which requires a password reset endpoint that does not exist yet.
- Fix approach: Add a `POST /v1/admin/users/{user_id}/reset-password` endpoint that generates a new temp password and emails it. Or surface the temp password in the API response (security tradeoff to document).

## Known Bugs

**`test.db` SQLite file is committed to git:**
- Symptoms: The test artifact `pco_backend/test.db` (77 KB) is tracked by git (`git ls-files` confirms). It is generated by pytest using `sqlite:///./test.db` (a file path, not `:memory:`).
- Files: `pco_backend/test.db`, `pco_backend/app/tests/conftest.py` (line 18)
- Trigger: Running `pytest` from `pco_backend/` creates/overwrites `test.db` on disk.
- Fix: Add `test.db` to `.gitignore`. Change `TEST_DATABASE_URL` in `conftest.py` to `"sqlite:///:memory:"` so no file is created.

**SQLite test dialect vs PostgreSQL production dialect mismatch:**
- Symptoms: Tests run against SQLite; production runs PostgreSQL. Several ORM models use `sqlalchemy.dialects.postgresql.UUID` which is a PostgreSQL-specific type. SQLite silently treats it as a string. JSON columns in `AuditLog` also behave differently across dialects.
- Files: `pco_backend/app/tests/conftest.py`, `pco_backend/app/models/user.py`, `pco_backend/app/models/event_pdf.py`, `pco_backend/app/models/audit_log.py`
- Trigger: UUID filtering, JSON queries, and timezone-aware datetime behavior can pass tests but fail in production.
- Fix approach: Switch test suite to use a real PostgreSQL instance (e.g., via Docker or `pytest-postgresql`) or use `testcontainers`. At minimum, document the dialect gap.

## Security Considerations

**No rate limiting on auth endpoints:**
- Risk: `POST /v1/auth/login` and `POST /v1/auth/refresh` have no rate limiting. A brute-force or credential-stuffing attack can submit unlimited login attempts.
- Files: `pco_backend/app/api/v1/auth.py`, `pco_backend/app/main.py`
- Current mitigation: Timing attack prevention via `_dummy_verify()` is present, but it does not limit volume.
- Recommendations: Add `slowapi` or a reverse-proxy-level rate limit on `/v1/auth/login`. Implement account lockout after N failed attempts.

**No logout / token revocation endpoint:**
- Risk: There is no `POST /v1/auth/logout` endpoint. Users cannot explicitly revoke a refresh token. If a device is lost or a token is compromised, the only options are waiting for expiry (30 days) or admin deactivating the account.
- Files: `pco_backend/app/api/v1/auth.py`
- Current mitigation: Admin deactivation revokes all tokens; token rotation on each refresh limits a stolen token's use to a single exchange.
- Recommendations: Add `POST /v1/auth/logout` that accepts the current refresh token and marks it revoked in the DB.

**No password change endpoint:**
- Risk: Members are assigned a temporary password on account creation and have no way to change it via the API. This means temp passwords may remain in use indefinitely.
- Files: `pco_backend/app/api/v1/auth.py`, `pco_backend/app/api/v1/users.py`
- Current mitigation: None.
- Recommendations: Add `POST /v1/users/me/change-password` accepting `current_password` and `new_password`.

**CORS allows all methods and all headers:**
- Risk: `app.add_middleware(CORSMiddleware, allow_methods=["*"], allow_headers=["*"])` is overly permissive. Non-browser clients are not restricted by CORS, but this signals incomplete hardening.
- Files: `pco_backend/app/main.py` (lines 55-61)
- Current mitigation: `allow_origins` is configurable via `CORS_ORIGINS` env var and defaults to `http://localhost:3000` only — this is the primary protection.
- Recommendations: Restrict `allow_methods` to `["GET", "POST", "PUT", "PATCH", "DELETE"]` and `allow_headers` to `["Authorization", "Content-Type"]`.

**Interest form has no spam or abuse protection:**
- Risk: `POST /v1/interest` is a public unauthenticated endpoint with no CAPTCHA, honeypot, or rate limiting. The only deduplication is a 409 on duplicate email, which is trivially bypassed with different email addresses.
- Files: `pco_backend/app/api/v1/interest.py`, `pco_backend/app/services/interest_service.py`
- Current mitigation: Email deduplication via unique index on `interest_submissions.email`.
- Recommendations: Add rate limiting per IP. Consider a CAPTCHA integration or a time-based honeypot field.

**Interest form exposes all submissions without pagination:**
- Risk: `GET /v1/interest` (admin only) returns all submissions as a flat list with no pagination. As submissions grow, this becomes a data dump in a single response.
- Files: `pco_backend/app/api/v1/interest.py`, `pco_backend/app/services/interest_service.py`
- Current mitigation: Admin-only access gate via `require_admin`.
- Recommendations: Add `skip`/`limit` pagination parameters.

**`SUPABASE_SERVICE_KEY` is a privileged credential with no scope restriction:**
- Risk: The Supabase service role key bypasses Row Level Security (RLS) and has full database access. If leaked, it grants full Supabase project access.
- Files: `pco_backend/app/storage/files.py`, `pco_backend/app/core/config.py`
- Current mitigation: Key is loaded from environment variable only; `.env` is in `.gitignore`.
- Recommendations: Scope the Supabase key to storage bucket access only if Supabase supports it. Rotate keys if exposed.

## Performance Bottlenecks

**Signed URL generation per event in a loop (N+1 external calls):**
- Problem: `list_events()` fetches all `EventPDF` rows, then calls `storage_service.create_signed_url()` for each one in a loop. Each call is a synchronous HTTP request to Supabase.
- Files: `pco_backend/app/services/event_service.py` (lines 17-28)
- Cause: No batching — Supabase Storage API signed URL generation is called once per event record.
- Improvement path: Cache signed URLs in the DB or a short-lived cache layer (e.g., Redis, in-process TTL cache) since URLs are valid for 3600 seconds. Alternatively, Supabase supports batch signed URL creation.

**No DB connection pool tuning:**
- Problem: `create_engine(settings.database_url, pool_pre_ping=True)` uses SQLAlchemy defaults (pool_size=5, max_overflow=10). No explicit pool configuration for the expected concurrency level.
- Files: `pco_backend/app/db/session.py`
- Cause: Not tuned for production load.
- Improvement path: Set explicit `pool_size`, `max_overflow`, and `pool_timeout` values based on the deployment environment and PostgreSQL connection limits.

## Fragile Areas

**Single-row tables with no enforcement:**
- Files: `pco_backend/app/models/rush_info.py`, `pco_backend/app/services/rush_service.py`
- Why fragile: `rush_info` is designed as a single-row table, but there is no DB-level constraint (unique index, check constraint, or trigger) preventing multiple rows. The service layer uses `.first()` to get the singleton, silently ignoring any extra rows.
- Safe modification: Always use `rush_service.get_rush()` / `rush_service.upsert_rush()` — never insert directly. Add a DB-level constraint to enforce the singleton if stability is critical.
- Test coverage: Tests cover the get/upsert behavior but do not test the multi-row edge case.

**Storage and DB writes are not atomic in event upload:**
- Files: `pco_backend/app/services/event_service.py` (lines 53-74)
- Why fragile: On `upload_event()`, the file is uploaded to Supabase Storage first, then a DB record is committed. If the DB commit fails, the code attempts to delete the orphaned file. But if that cleanup also fails (line 69: `pass`), a file is left in storage with no DB record.
- Safe modification: Ensure the cleanup failure is at minimum logged (currently it is silently swallowed). Consider a periodic reconciliation job to detect orphaned storage files.
- Test coverage: `test_upload_db_failure_triggers_storage_delete` tests the rollback path but does not test the double-failure case (DB fails + storage cleanup fails).

**Expired refresh tokens accumulate in the DB:**
- Files: `pco_backend/app/models/refresh_token.py`, `pco_backend/app/api/v1/auth.py`
- Why fragile: Expired and revoked refresh tokens are never deleted. Every login creates a new row; every refresh creates a new row and marks the old one revoked. Over time the `refresh_tokens` table grows unboundedly.
- Safe modification: Add a periodic cleanup job or a DB cron to delete rows where `revoked = true OR expires_at < now()`.
- Test coverage: Not tested.

## Scaling Limits

**Single Alembic migration for all tables:**
- Current state: One migration file (`001_initial_schema.py`) creates all 7 tables. There are no incremental migrations.
- Limit: Any schema change requires writing a new migration. If initial migration is modified instead, existing deployments with `alembic_version=001` will not re-run it, causing silent schema drift.
- Scaling path: Treat migrations as append-only. Each schema change gets its own numbered migration file.

**No pagination on any list endpoint:**
- Current state: `list_members()`, `list_events()`, `list_submissions()`, and `get_leadership()` all return full result sets with no pagination.
- Files: `pco_backend/app/services/user_service.py`, `pco_backend/app/services/event_service.py`, `pco_backend/app/services/interest_service.py`, `pco_backend/app/api/v1/content.py`
- Limit: Response size and query time grow linearly with data volume. No current enforcement of result caps.
- Scaling path: Add `skip: int = 0` / `limit: int = 100` query parameters with a max cap to all list endpoints.

## Dependencies at Risk

**`next` version 16.1.6 is non-standard / likely pre-release:**
- Risk: The frontend `package.json` pins `next` at `16.1.6`. As of early 2026, Next.js stable is in the 15.x line. Version 16.x may be a pre-release or canary build, which can be unstable and unsupported.
- Files: `pco_website/package.json`
- Impact: Pre-release frameworks may have breaking changes between minor versions and limited community support.
- Migration plan: Audit whether `16.1.6` is an intentional canary pin. If not, downgrade to the latest stable `15.x` release.

## Missing Critical Features

**No password change or reset flow:**
- Problem: Users receive a temporary password via email on account creation. There is no API endpoint to change it and no forgot-password/reset flow.
- Blocks: Any member who loses their temp password, forgets their set password, or whose email delivery failed has no self-service recovery path.

**No logout endpoint:**
- Problem: There is no `POST /v1/auth/logout` to explicitly revoke a refresh token.
- Blocks: Secure session termination from the frontend. Users cannot log out across devices.

**`/v1/public/info` endpoint is not connected to DB:**
- Problem: The public info endpoint hardcodes `"TODO"` for all fields rather than reading from `OrgContent` and `RushInfo` tables that already exist.
- Blocks: Any public-facing page that needs org info, rush dates, or contact details.
